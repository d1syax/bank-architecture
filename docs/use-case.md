# Use Cases: Autistic Bank API

## Реєстрація користувача

**Актор:** Неавторизований користувач

**Передумови:** Відсутні

**Основний сценарій:**
1. Гість відправляє `POST`-запит з email, паролем та ПІБ.
2. Система валідує формат email та перевіряє чи не зайнятий він.
3. Система хешує пароль та створює запис у базі даних.
4. Система повертає статус `200 OK` та JWT-токен.

**Альтернативні сценарії / помилки:**
* Невалідний формат email або порожній пароль: `400 Bad Request`
* Користувач з таким email вже існує: `409 Conflict`

---

## Вхід користувача

**Актор:** Неавторизований користувач

**Передумови:** Користувач має існуючий акаунт

**Основний сценарій:**
1. Користувач відправляє `POST`-запит з email та паролем.
2. Система знаходить користувача та порівнює пароль з хешем.
3. Система повертає статус `200 OK` та JWT-токен.

**Альтернативні сценарії / помилки:**
* Email не знайдено або пароль невірний: `401 Unauthorized`
* Відсутні обов'язкові поля: `400 Bad Request`

---

## Створення банківського рахунку

**Актор:** Авторизований користувач

**Передумови:** Користувач успішно автентифікований

**Основний сценарій:**
1. Користувач відправляє `POST`-запит із зазначенням валюти (USD, UAH, EUR).
2. Система перевіряє JWT-токен.
3. Система генерує унікальний номер рахунку та встановлює баланс 0.
4. Система повертає статус `201 Created` та дані рахунку.

**Альтернативні сценарії / помилки:**
* Відсутній або невалідний токен: `401 Unauthorized`
* Невідома валюта: `400 Bad Request`

---

## Перегляд рахунків

**Актор:** Авторизований користувач

**Передумови:** Користувач успішно автентифікований

**Основний сценарій:**
1. Користувач відправляє `GET`-запит.
2. Система перевіряє JWT-токен.
3. Система повертає список рахунків поточного користувача зі статусом `200 OK`.

**Альтернативні сценарії / помилки:**
* Відсутній або невалідний токен: `401 Unauthorized`

---

## Переказ коштів

**Актор:** Авторизований користувач

**Передумови:** Користувач має рахунок з достатнім балансом

**Основний сценарій:**
1. Користувач відправляє `POST`-запит з ID рахунку відправника, ID рахунку отримувача та сумою.
2. Система перевіряє JWT-токен.
3. Система перевіряє існування обох рахунків.
4. Система перевіряє інваріант: сума > 0.
5. Система перевіряє інваріант: баланс відправника >= сума переказу.
6. Система виконує переказ в рамках однієї транзакції БД.
7. Система повертає статус `200 OK`.

**Альтернативні сценарії / помилки:**
* Відсутній або невалідний токен: `401 Unauthorized`
* Сума <= 0: `400 Bad Request`
* Один із рахунків не знайдено: `404 Not Found`
* Недостатньо коштів: `400 Bad Request`

## Поповнення рахунку

**Актор:** Авторизований користувач

**Передумови:** Користувач має існуючий рахунок

**Основний сценарій:**
1. Користувач відправляє `POST`-запит з ID рахунку та сумою поповнення.
2. Система перевіряє JWT-токен.
3. Система перевіряє інваріант: сума > 0.
4. Система додає кошти до рахунку.
5. Система повертає статус `200 OK`.

**Альтернативні сценарії / помилки:**
* Відсутній або невалідний токен: `401 Unauthorized`
* Рахунок не знайдено: `404 Not Found`
* Сума <= 0: `400 Bad Request`
